# 情境一：活動網頁百倍流量

核心思路是**事前準備重於事後應急**，流量衝擊是可預期的，活動前就要處理完。

## 容量規劃與壓力測試

先用 k6 或 Locust 模擬到預期峰值的 1.5 倍，找出哪個環節先撐不住。通常瓶頸會在 DB connection pool 或沒加快取的 API，不一定是 CPU。

壓測時重點觀察 P99 Response Time、Error Rate、DB 連線數、Cache Hit Rate。

## 基礎設施調整

**CDN**：靜態資源走 CloudFront，如果活動頁內容固定，直接全頁靜態化，Origin 完全不用動。

**擴容**：不要只依賴 HPA，活動開始瞬間是尖峰，等 HPA 反應已經來不及了，活動前就預先擴到預估規模。

**DB**：加 Read Replica 分擔讀取，查詢加 Redis 快取。特別要注意 **connection pool 預熱**，活動開始前就先建好 pool，不然第一波流量進來時 pool 從零開始建，會有一批請求先失敗。

**流量削峰**：報名、抽獎這類非即時操作丟進 SQS 非同步處理，不要讓使用者每次點擊都直接打到 DB。

**WAF + Rate Limiting** 也要設好，百倍流量也會吸引惡意流量和爬蟲一起進來。

## 活動期間

Grafana 監控 P50/P95/P99、Error Rate、DB 連線數，告警接 Slack，安排 SRE 值班。

準備好 Feature Flag，萬一撐不住可以快速關掉推薦、個人化等非核心功能，確保主流程能跑。事前確認可以一鍵回滾到上一個穩定版本，並備好緊急聯絡清單。
